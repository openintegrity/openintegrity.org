Open Integrity Index Technology Roadmap

This document provides an overview of how various aspects of the Open Integrity Index technology components will evolve. The aim is to propose a logical progression from basic capabilities to more advanced ones, based on existing data and partnerships (Libraries.io, TOSDR, Ranking Digital Rights), and supporting the objectives of the Open Integrity Index.

The overall priorities will progress as follow:

  • Develop a data model: starting with documentation, then formalise a schema to enable a decentralised architecture.

  • Collect existing metrics and develop new metrics: start with basic open source project data, move towards security related data as well as legal and measurable service data.

  • Make data easy to consume: starting with bulk download, then simple query API and moving toward a federated query API.

These priorities will evolve based on the consultation with data consumer partners. In addition other low hanging fruits will be looked at such as:

  • Help define a unique identifier for projects

  • Help standardize document update monitoring (TOSBack)

At a later stage more complex querying, collection of data to feed an index cache, data streaming API, tools for validating measurement partner schema adoption. Also see:

Roadmap

Metrics roadmap

Measurement Agent or Measurement Partner Agent in -> Metrics out

OII metrics are metrics which are made available through the Open Integrity Index metrics API. It can be collected through measurement partners or by OII itself. See Metrics spreadsheet. Specifically the** oii-metrics version **column.

The metrics repo should include the measurement partners' measurement agent's code and include OII's own measurement agents as well as deployment scripts (fabfiles, ansible scripts, dockerfiles,...).

The following roadmap items correspond to increase of overall coverage of metrics available via the Open Integrity Initiative. Depending on partnership progress and research, different metrics might be developed. For instance,

oii-metrics-0.1.0

Basic Project metrics.

  • Project relevance metrics. (For instance, oii-metrics-relevance-google-trends-per-country)

  • Project identification metrics (to help uniquely identify projects)

  • Libraries.io (existing data, using oii-schema-0.1.0)

These metrics will typically be collected by "Download / Parse / Update" Agents which will access a URL, parse it, perform a calculation or transformation, and update a metric repository.

Agents

Open Integrity might develop Measurement Agent code and operate Agents when it fills a gap and helps downstream data consumers. One example is with regards to monitoring web page changes and keeping track of changes. This type of agent could be reused by TOSBack for monitoring changes to terms of service, as well as Libraries for monitoring README changes or CHANGELOG changes.

It doesn't make sense to impose a technology stack for Agents, they should however adhere to a particular API format for metric submission which will include a schema.

  • Measurement partners should be able to add or register a Metric.

    • It should be either an agreed Metric Type that is described in the master OII Metric Taxonomy, if not the new Metric Type should be submitted to the Consultation group.

    • It should either refer to an existing project (with a unique identifier)

Analytics roadmap

Data Storage & Analytics layer.

Analytics capability will be mainly drawn from partnerships in the future. At this stage, simple capabilities will be provided in order to query the metrics repository and data catalogue.

Ultimately, how expressive does the API need to be? In particular, should it allow getting large time series or also do some custom querying

Providing analysis of code will depend on the approach taken. It will require either a batch or a streaming data processing pipeline or a combination of both ("lambda architecture"). It might be most cost efficient and faster to launch to make use of existing cloud computing services such as Big Query, Cloud DataFlow,RedShift or others. Partnerships with existing software repository mining projects might also be pursued (such as with BOA - boa.cs.iastate.edu).

Data governance requirements (for reproducibility and availability) might require to allow the deployment of data processing systems also on private clouds. In addition, given the requirements for storing data and the network bandwidth required to periodically update metrics, it might be more sustainable to deploy a private cloud and use existing open source orchestration systems combined with big data cluster management software (such as Ambari) to provision, manage and monitor the metrics collection infrastructure. Fortunately, a number of Open Integrity Initiative potential partners (such as Libraries.io, Google, M-Lab) and iilab strategic partners (such as Greenhost, OONI) already manage similar systems. In addition, with increasing visibility and partnerships, it will be possible to follow a similar approach to that of open source operating systems distributions and manage a mirror program.

Storage & Bandwidth

A number of metrics will depend on the existence of historical data, which will increase the storage requirements, as well as depend on frequent pulling of updated data, which will also increase network bandwidth requirements.

It might be preferable to only store raw project data based on sets of criteria, such as the pre-existence of mirrors or importance indicators (number of depending code bases, audience and usage).

Such raw data would include code, the content of issue trackers and support forums, as well as web pages (containing documentation, manuals, policies, licenses…).

The infrastructure will need to support the mirroring of code repositories. There are estimates that there are 50 to 100 billion lines of code in active service and others that estimate up to a trillion. Lammel, R., & Verhoef, C. (2001). Cracking the 500-language problem. Software, IEEE, 18(6), 78-88. This can be roughly estimated to amount to 100 Terabytes. Probably only a small percentage of this code is accessible and therefore storage of all possible code is doable at a reasonable cost.

Depending on the way the code is accessible (git, svn, cvs,...) there will be an overhead for the repository metadata. It should be possible to optimise the download agents to avoid the download of raw data for large binary blobs (most likely media files).

The storage and bandwidth necessary for the verification and monitoring of web pages containing textual data (such as terms of service, privacy policies, documentation,...). The tosback-data repository is a dataset of 888+ privacy policies & terms of service and amounts to around 1GB.

Raw Metrics User Personas

In order to help design the Open Integrity metrics repository and API the following preliminary user personas have been developed. During the Inception, Framework, Data and Outreach phase, the personas will be further developed.

Project Owner

  • Developer of a tool wanting to track themselves.

End-User Advisor / Analyst

  • Analyst trying to put together scores to support decision making of end-users or provide downstream calculated data that can be used as advice. i.e. EFF Secure Messaging Scorecard…

  • CII Census,

Researcher / Analyst

  • Analyst wanting to answer research questions, possibly within an academic setting.

  • Example of research questions:

    • "how people perceive and consider the potential impacts of their own and others’ edits as they write together? [Dourish and Bellotti 1992]"; — “what is the most widely used open source license? [Lerner and Tirole 2002]”; “how many projects continue to use DES (considered insecure) encryption standards? [Landau 2000]”; “how many open source projects have a restricted export control policy? [Goodman et al. 1995]”; — “how many projects on an average start with an existing code base from another project instead of scratch? [Raymond 1999]”; — “how often do practitioners use dynamic features of JavaScript, e.g. eval? [Richards et al. 2011]”; and — “what is the average time to resolve a bug reported as critical? [Weiss et al. 2007]”. (http://www.cs.bgsu.edu/rdyer/papers/tosem15.pdf)
