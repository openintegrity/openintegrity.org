Design Practice

The Design practices identified at the metrics workshop are listed below. Follow links for details.

User Experience Frustrations

Good practice

  • Clear data disposal policy from the outset
  • Allowing users to know where to find trusted expertise and tool comparisons
  • Use "carrots" not just "sticks"

OK practice

  • Tension between wanting to respond to user frustrations and not wanting to collect personally identifiable information
  • Surprises
  • "TL;DR" - Too much information for users to realistically consume
  • Never-ending "wizards"
  • Encouraging users to click "Next" without understanding

Poor practice

  • Paste from clipboard prevented during authentication
  • Requiring password change too frequently

Ease of Use

Good practice

  • Different interfaces for beginner and advanced users
  • Error messages explain what the problem is (in case it relates to user action)
  • Language appropriate for beginners
  • High quality user support
  • Clear description of the result of user choices
  • Recommended settings and/or configuration templates
  • Cross-platform consistency
  • Configuration history (defaults, undo, redo, etc.)

User Education

Good practice

  • In-app guides
  • Clear "introduction"
  • Live chat support
  • Easy path to the right docs
  • Explainations of user interface elements
  • Tool tips
  • Downloadable guides for offline use
  • Explanation of how to get help
  • Multiple levels of guides for each topic (expand/collapse)
  • A range of threat models and user levels

OK practice

  • Multiple-langauge translations
  • Iconographic/pictoral guides that do not require translation
  • Advanced guide for power users
  • "Learn more" buttons

Poor practice

  • Confusing icons
