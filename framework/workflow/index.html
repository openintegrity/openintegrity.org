Impact

  • Relevant Impact Group (for instance partners knowledgeable about and interested in Iran) get notified #5

  • Consensus on Best Practices -> Practice (decision) #8

  • Discussion of potential measurement approaches -> Metrics (measurement)
  • Development of collection methodologies or collection agent proof-of-concepts -> Metrics (collection) #10

Data

Getting to the layers behind a single data point

Data Consumer

From Data point to Impact story**

  • Data Consumer looks up Skype
  • They see a data point on 2FA availability which says that it is "only available with a Microsoft account".
  • They see that the data point has been reviewed by a number of individuals/organisations the data consumer trusts.
  • They can click on a "?" icon to get more information about the context of this particular data point such as:
  • Measurement Metadata:
    • Source (which, which collection approach, verification approach,...)
    • Collection (date, collecting agent,...)
  • Metric Metadata:
    • Explanation: How is this particular data point correlated to the adoption of 2FA for this tool?
    • Metric Type: It could be that it's an expert review, a user review, has a screenshot or a web page capture as evidence, from one or many network vantage points - offering different correlation strength.
  • Practice Metadata:
    • Definition: How is 2FA understood? What is the scope of our definition for it.
    • Mitigation: Which attack pattern does it mitigate and how? <- this is a result of the Practice (decision) framework process.
  • Impact Metadata
    • Pattern: What is the alice and bob description of the attack pattern.
    • Stories: Which reported stories fall under this attack pattern category.

Review Triggers

A number of data related events could trigger the need for a review. Here are some example of event types (which could all be subscribed to independently, or by different groups of users)

  • A new tool has been submitted
  • The underlying source has changed (like a watched web page) making a data point "stale"
  • A dispute occurs.
  • The overall coverage or freshness metrics are below a certain threshold.
  • A bounty is made available.

There should be clear signage about:

  • the type of expertise needed to contribute to this particular trigger
  • maybe the time it will take to contribute

Browsing Reviewer

They visit the site because they've heard about it and are curious. There should be a number of call for action on the page:

  • a few "Join" or "Contribute" buttons leading to
    • The current [join](page
  • stale/disputed/empty data points should be marked and have a call to action (including to put a bounty on something that someone would like)

Engaged Reviewer

A user, another reviewer or the platform's social media feed create engagement with potential reviewers.

  • They might receive a social media alert that a data point needs review on the platform.
  • They should land on a page which show the relevant data point and:
    • Describes the actions that need to be undertaken (allowing to click through to do the work)
    • Surface a few other potential tasks (allowing to engage others in their network that might want to help)
    • They might be able to "Grab" it - to show someone is working on it (possibly publicly)

Subscribed Reviewer

They have registered as a reviewer and they can configure their notifications to receive:

  • regularly (immediately, daily, weekly)
  • a number of (one, batch of 5 or 10)
  • specific event type/review triggers
