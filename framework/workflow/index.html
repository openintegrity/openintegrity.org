Collaboration workflows target these categories of users:

  • Impact reporters
  • Practice debaters
  • Metrics developers / Measurement partners
  • Data contributors or reviewers

Trying to reuse existing collaboration infrastructure capabilities on Gitlab in order to minimise the development of workflow software or use of dedicated software.

Framework: Taking stock of the evolution of the Threat Landscape

Framework User: From Impact story to Measurement Data

Impact

  • User submits new story about Iran 2FA hacking -> Impact (partner) #1 #2 #4
  • Relevant Impact Group users get notified #5
  • Verification -> Impact (review) #3
  • New attack pattern -> Impact (analysis) #3 #6

Practice

  • Browsing of relevant issues that need attention #7
  • Notification of Communities of Practice #8
  • Debate Mitigation techniques (across multiple expertises) -> Practice (discuss) #7
  • Consensus on Best Practices -> Practice (decision) #8

Metrics

  • Discussion on metrics for Best Practice -> Metrics
  • Discussion of potential measurement approaches -> Metrics (measurement)
  • Development of collection methodologies or collection agent proof-of-concepts -> Metrics (collection) #10

Framework: Community of Practice participants

Eg. Second Muse

  • Aims are to explore the issues and understand at a glance what are the type of conversations taking place, what needs attention, and things that might be worth following or engaging on.
  • OII Team shares a link to the larger community of practice "category" entry point /practices/design/ ?)
  • The page explains the shared key principles of the Framework ( : [](/framework/index.short)) and of the Community of Practice approach : [](/practices/index.short and goes into more details of the interaction between this particular community of practice (what does "Design" mean here) and Open Integrity (why is design important to user security and privacy)
  • The call to action should be ordered from the most general need to the most specific (where as on the webapp it should probably be reversed)
    • Help organise the Design Practice ongoing consultation
    • Participate in conversations about
      • User Experience Best Practice for Security and Privacy
        • Surface info about:
          • Current thread tagged for "review?"
          • Participation metrics? (Could be from the RightsCon session)
          • A current theme?
      • Translation and Localisation Best Practices for Security and Privacy
      • Visual Language and Iconography Best Practices for Security and Privacy
  • After clicking on the "User Experience" Community of Practice link, the user should see a page that's grabbing data from the issue tracker that could be a cross between a Discourse Topic page, a Github Issue list and a Wiki (Also see How to Build a Collective Intelligence Platform to Crowdsource Almost Anything)

Practice lifecycle

e.g

  • Practice : Tech - Infrastructure "An internal registry should be setup which rebuilds containers from source."
  • [When scaling add a pre-moderation step with a notification to a moderator group]
  • This should alert subscribed CoP participants. Including
    • Who submitted this.
    • A short title.
    • A call to action. ("Review")
  • There should be some rules for when Practices are moving to the next step.
