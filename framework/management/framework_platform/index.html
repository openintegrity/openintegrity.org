Framework Platform

User Stories

As a researcher in the Impact Group I can add and tag my desk research on known incidents with a particular tool or within a particular community

As a member of the Impact Group who works closely with at-risk users, I have a place to record and tag anonymized digital security incidents and vulnerabilities as I learn about them, even if I do not have all of the details

As a security techie in the Impact Group, I can add technical clarification and link to related research after learning of a new incident or vulnerability that has been added to the platform but that lacks details

As a member of the Impact Group, I can link to and tag articles and reports that I think might be relevant to existing incidents or vulnerabilities on OII

As a member of the Impact Group, I can easily learn of new additions to the platform that are relevant to my expertise or to the communities with which I work

Approach

  • Iterate. It should be easy to draft content including folder structure, with loosely linked, sparse files.
  • Release early and often. There should be minimal barriers to releasing work in progress.
  • Clear labeling. It should be easy for writers and readers to differentiate draft work from more mature work. A la commitizen/conventional-changelog.
  • Small repos. As in code, try to separate concerns of content repos.

Problems to be mindful of:

  • Divergence: lots of sparse, unconnected files. -> TOC/TOC Crosslinks
  • Calcification: structures don't evolve because of the cost of refactoring repos/folder/files.

Workflow

  • Draft/Published
  • Draft/Ready for Review/In Review/Reviewed/...

  • .../RTBC?

  • Are there private spaces for elaborating details with partners for instance?

Platform

Evaluation of assembl vs a content as code approach.

Key features:

  • For free (versioning, branches, pull requests)
  • Discussion (via code review / github's ?)
  • Harvesting/Synthesis
  • Traceability (via some form of validated semantic linking)

Nice to have

  • Personal annotations.

Managing comments in a code review tool:

  • pros: separation of concerns.
  • cons: comments are not in the markdown (so can't be checked out).
  • mitigation: output commented version (generate diffs, tooltips and medium style side comments). Roundtripping that would be hard (i.e. adding comments by committing changes to the markdown versions) but probably doable. We're back to using Markdown as a persistence format.
